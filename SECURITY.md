# Security policy

## Supported versions

Security fixes are applied on the default branch and released through the normal release process for each Twilic implementation repository. Older release lines receive fixes when maintainers still support them; check the repository README or release notes for details.

## Reporting a vulnerability

**Please do not open a public GitHub issue for security vulnerabilities.**

Report vulnerabilities through [GitHub private vulnerability reporting](https://docs.github.com/en/code-security/security-advisories/guidance-on-reporting-and-writing-information-about-vulnerabilities/privately-reporting-a-security-vulnerability) on the affected repository:

1. Open the repository on GitHub.
2. Go to **Security** → **Report a vulnerability** (or use the **Security** tab → **Advisories** → **Report a vulnerability**).

If private reporting is unavailable on a given repository, contact the maintainers through the [Twilic organization](https://github.com/twilic) and ask for a private channel before sharing exploit details publicly.

Include, when possible:

- A description of the issue and impact
- Steps to reproduce or a proof of concept
- Affected versions, components, or repositories
- Any suggested fix or mitigation

## Response expectations

| Stage                              | Target                                               |
| ---------------------------------- | ---------------------------------------------------- |
| Initial acknowledgment             | Within **5 business days**                           |
| Severity assessment and next steps | Within **10 business days**                          |
| Fix or documented mitigation       | Depends on severity; critical issues are prioritized |

We may request additional information and will coordinate disclosure timing with reporters when credit or embargo is requested.

## Safe harbor

We appreciate responsible disclosure. Reporters acting in good faith who follow this policy will not be asked to cease testing or reporting solely because they discovered a vulnerability, provided they avoid privacy violations, data destruction, and service disruption beyond what is necessary to demonstrate the issue.
